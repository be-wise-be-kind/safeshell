Architecture: Daemon-Based Execution

IMPLEMENTED   Eliminating Python startup overhead by moving execution into the daemon

1. The Problem (Solved)

Previous Performance Issue

Every command intercepted by SafeShell used to pay a ~250ms Python startup penalty, even though we had a warm daemon running.

Python startup: ~100ms Module imports: ~150ms Total overhead: ~250ms per command

Old Architecture Flow (Before)

sequenceDiagram
    participant T as Terminal
    participant S as Shim (bash)
    participant W as Wrapper (Python)
    participant D as Daemon (Python)
    participant SH as Real Shell

    Note over T,SH: User types "ls -la"

    T->>S: ls -la
    S->>W: spawn new Python process
    Note over W: 250ms startup penalty!
    W->>W: import safeshell.* modules
    W->>D: evaluate("ls -la")
    D->>D: check rules (fast, ~1ms)
    D-->>W: allow
    W->>SH: execute "ls -la"
    SH-->>W: output + exit code
    W-->>T: output
    W->>W: process exits

    Note over T,SH: User types "git status"

    T->>S: git status
    S->>W: spawn ANOTHER new Python process
    Note over W: 250ms penalty AGAIN!
    W->>W: import safeshell.* modules AGAIN
    W->>D: evaluate("git status")
    D-->>W: allow
    W->>SH: execute
    SH-->>W: output
    W-->>T: output

Why This Happened

The daemon was warm and fast - it already had all Python modules loaded. But the wrapper client was a separate Python process that:

  1. Spawned fresh for every single command
  2. Re-imported all modules (pydantic, loguru, safeshell.*)
  3. Connected to daemon, got response
  4. Executed command and exited
  5. Repeated this entire cycle for the next command

The daemon's warmth was wasted because we paid Python startup cost on every command anyway.

2. Solution: Daemon-Based Execution

Key Insight

Move command execution INTO the daemon. The daemon already has Python warm - let it evaluate AND execute commands. The shim becomes a simple socket client (no Python needed).

Socket connection: ~0.5ms Daemon evaluation: ~0.5ms Fork + exec: ~0.5ms Total overhead: ~1.5ms per command

Achieved Performance: 1000 commands in ~1.5 seconds (~1.5ms each) - 100x+ improvement

Current Architecture Flow

sequenceDiagram
    participant T as Terminal
    participant S as Shim (bash + socat)
    participant D as Daemon (Python, warm)
    participant C as Child Process
    participant SH as Real Shell

    Note over T,SH: User types "ls -la"

    T->>S: ls -la
    S->>D: socket: execute("ls -la")
    Note over D: Already warm, no startup!
    D->>D: check rules (~1ms)
    D->>C: fork()
    C->>SH: exec("ls -la")
    SH-->>C: output + exit code
    C-->>D: results via pipe
    D-->>S: socket: output + exit code
    S-->>T: output

    Note over T,SH: User types "git status"

    T->>S: git status
    S->>D: socket: execute("git status")
    Note over D: Still warm!
    D->>D: check rules (~1ms)
    D->>C: fork()
    C->>SH: exec("git status")
    SH-->>C: output
    C-->>D: results
    D-->>S: output + exit code
    S-->>T: output

5. Protocol Changes

New Request Type: execute

Add a new request type that combines evaluation and execution:

{
    "type": "execute",
    "command": "ls -la",
    "working_dir": "/home/user/project",
    "env": {"PATH": "...", "HOME": "..."},
    "execution_context": "human"
}

New Response Format

Response includes execution results:

{
    "success": true,
    "decision": "allow",
    "executed": true,
    "exit_code": 0,
    "stdout": "file1.txt\nfile2.txt\n",
    "stderr": "",
    "execution_time_ms": 12
}

For denied commands:

{
    "success": true,
    "decision": "deny",
    "executed": false,
    "denial_message": "Cannot commit directly to protected branch"
}

6 Wrapper Keep Python wrapper as fallback for complex cases (approval UI, etc.)

7. Risks and Mitigations

Risk: Daemon becomes single point of failure

Mitigation: Keep fail-open behavior. If daemon is unreachable, shim falls back to direct execution (configurable).

Risk: Fork overhead for each command

Mitigation: Fork is fast (~5ms on Linux). Much faster than Python startup (~250ms). Can optimize further with process pooling if needed.

Risk: Large stdout/stderr transfers

Mitigation: Stream output back via socket rather than buffering. Set reasonable limits. Consider passing file descriptors directly for very large outputs.

Risk: Signal handling complexity

Mitigation: Child process runs in its own process group. Parent daemon handles SIGCHLD. Ctrl+C in terminal sends to child's process group.

8. Design Decisions

  1. TTY handling: Non-interactive only. Buffered stdout/stderr. Interactive commands can fall back to direct execution if needed.
  2. Environment inheritance: Child receives env from shim request, merged with current env (PATH preserved).
  3. Working directory: Child uses subprocess with cwd parameter to run in requested working_dir.
  4. Streaming vs buffered: Buffered output returned at end (simpler, sufficient for most commands).
  5. Timeout handling: No timeout - daemon should be non-invasive and not change command behavior.
  6. Builtin overrides: Configurable via check_cd, check_source, check_eval settings. Eval disabled by default due to shell hook overhead.
