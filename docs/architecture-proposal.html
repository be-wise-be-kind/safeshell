Architecture Proposal: Daemon-Based Execution

DRAFT - Pending Review   Eliminating Python startup overhead by moving execution into the daemon

1. The Problem

Current Performance Issue

Every command intercepted by SafeShell pays a ~250ms Python startup penalty, even though we have a warm daemon running.

Python startup: ~100ms Module imports: ~150ms Total overhead: ~250ms per command

Current Architecture Flow

sequenceDiagram
    participant T as Terminal
    participant S as Shim (bash)
    participant W as Wrapper (Python)
    participant D as Daemon (Python)
    participant SH as Real Shell

    Note over T,SH: User types "ls -la"

    T->>S: ls -la
    S->>W: spawn new Python process
    Note over W: 250ms startup penalty!
    W->>W: import safeshell.* modules
    W->>D: evaluate("ls -la")
    D->>D: check rules (fast, ~1ms)
    D-->>W: allow
    W->>SH: execute "ls -la"
    SH-->>W: output + exit code
    W-->>T: output
    W->>W: process exits

    Note over T,SH: User types "git status"

    T->>S: git status
    S->>W: spawn ANOTHER new Python process
    Note over W: 250ms penalty AGAIN!
    W->>W: import safeshell.* modules AGAIN
    W->>D: evaluate("git status")
    D-->>W: allow
    W->>SH: execute
    SH-->>W: output
    W-->>T: output

Why This Happens

The daemon is warm and fast - it already has all Python modules loaded. But the wrapper client is a separate Python process that:

  1. Spawns fresh for every single command
  2. Re-imports all modules (pydantic, loguru, safeshell.*)
  3. Connects to daemon, gets response
  4. Executes command and exits
  5. Repeats this entire cycle for the next command

The daemon's warmth is wasted because we pay Python startup cost on every command anyway.

2. Proposed Solution: Daemon-Based Execution

Key Insight

Move command execution INTO the daemon. The daemon already has Python warm - let it evaluate AND execute commands. The shim becomes a simple socket client (no Python needed).

Socket connection: ~1ms Daemon evaluation: ~1ms Fork + exec: ~5ms Total overhead: ~10ms per command

Proposed Architecture Flow

sequenceDiagram
    participant T as Terminal
    participant S as Shim (bash + socat)
    participant D as Daemon (Python, warm)
    participant C as Child Process
    participant SH as Real Shell

    Note over T,SH: User types "ls -la"

    T->>S: ls -la
    S->>D: socket: execute("ls -la")
    Note over D: Already warm, no startup!
    D->>D: check rules (~1ms)
    D->>C: fork()
    C->>SH: exec("ls -la")
    SH-->>C: output + exit code
    C-->>D: results via pipe
    D-->>S: socket: output + exit code
    S-->>T: output

    Note over T,SH: User types "git status"

    T->>S: git status
    S->>D: socket: execute("git status")
    Note over D: Still warm!
    D->>D: check rules (~1ms)
    D->>C: fork()
    C->>SH: exec("git status")
    SH-->>C: output
    C-->>D: results
    D-->>S: output + exit code
    S-->>T: output

5. Protocol Changes

New Request Type: execute

Add a new request type that combines evaluation and execution:

{
    "type": "execute",
    "command": "ls -la",
    "working_dir": "/home/user/project",
    "env": {"PATH": "...", "HOME": "..."},
    "execution_context": "human"
}

New Response Format

Response includes execution results:

{
    "success": true,
    "decision": "allow",
    "executed": true,
    "exit_code": 0,
    "stdout": "file1.txt\nfile2.txt\n",
    "stderr": "",
    "execution_time_ms": 12
}

For denied commands:

{
    "success": true,
    "decision": "deny",
    "executed": false,
    "denial_message": "Cannot commit directly to protected branch"
}

6. Implementation Plan

Step Component Changes
1 Protocol Add RequestType.EXECUTE and response fields for stdout/stderr/exit_code
2 Daemon Add _handle_execute() that evaluates then forks to run command
3 Executor New module daemon/executor.py - fork, capture output, return results
4 Shim Client Update safeshell-check bash script to use new protocol
5 Shims Update safeshell-shim and init.bash to use bash client
6 Wrapper Keep Python wrapper as fallback for complex cases (approval UI, etc.)

7. Risks and Mitigations

Risk: Daemon becomes single point of failure

Mitigation: Keep fail-open behavior. If daemon is unreachable, shim falls back to direct execution (configurable).

Risk: Fork overhead for each command

Mitigation: Fork is fast (~5ms on Linux). Much faster than Python startup (~250ms). Can optimize further with process pooling if needed.

Risk: Large stdout/stderr transfers

Mitigation: Stream output back via socket rather than buffering. Set reasonable limits. Consider passing file descriptors directly for very large outputs.

Risk: Signal handling complexity

Mitigation: Child process runs in its own process group. Parent daemon handles SIGCHLD. Ctrl+C in terminal sends to child's process group.

8. Open Questions

  1. TTY handling: Should daemon allocate a PTY for interactive commands? Or only support non-interactive?
  2. Environment inheritance: Should child inherit daemon's env or receive env from shim request?
  3. Working directory: Child should chdir to requested working_dir before exec - any edge cases?
  4. Streaming vs buffered: Stream output in real-time or buffer and return at end?
  5. Timeout handling: Should daemon kill long-running commands? Configurable timeout?
